# OpenClaw 安全加固

## 1. 默认安全机制

OpenClaw 在设计上遵循 **安全优先** 原则，默认配置即具备多层防护：

| 机制 | 说明 | 默认值 |
|------|------|--------|
| **Gateway 绑定** | 仅监听本地回环地址，不对外暴露 | `127.0.0.1:18789` |
| **配对认证** | DM 消息需要配对码审批后才能交互 | `dmPolicy: "pairing"` |
| **Auth Token** | Gateway API 调用需携带认证令牌 | onboard 时自动生成 |
| **沙箱隔离** | 非主会话在 Docker 容器中执行工具 | `sandbox.mode: "non-main"` |
| **只读根文件系统** | 沙箱容器禁止修改系统文件 | `readOnlyRoot: true` |
| **网络隔离** | 沙箱容器默认无网络访问 | `network: "none"` |
| **资源限制** | 沙箱容器的内存和 CPU 受限 | `memory: "1g"`, `cpus: 1` |

## 2. 目录访问控制

### 2.1 Agent 可访问的目录

Agent 的活动范围受 Workspace 配置约束。以下目录是 Agent 正常工作所需的可访问范围：

```
~/.openclaw/
├── workspace/                 # ✅ Agent 主工作目录
│   ├── AGENTS.md              #    操作指令（每次会话加载）
│   ├── SOUL.md                #    人格和行为边界
│   ├── USER.md                #    用户信息和偏好
│   ├── IDENTITY.md            #    Agent 名称和特征
│   ├── TOOLS.md               #    本地工具说明
│   ├── HEARTBEAT.md           #    心跳定时任务清单
│   ├── BOOT.md                #    启动仪式脚本
│   ├── MEMORY.md              #    长期记忆索引
│   ├── memory/                #    日记忆文件
│   ├── skills/                #    工作区技能
│   └── canvas/                #    UI 文件
├── sessions/                  # ✅ 会话记录（Agent 上下文）
└── openclaw.json              # ⚠️ 主配置文件（Agent 可读取非敏感配置）
```

**沙箱中的目录挂载**：当 Agent 在 Docker 沙箱中执行工具时，只有 Workspace 目录被挂载进容器，且可通过 `workspaceAccess` 控制读写权限：

```json5
{
  "sandbox": {
    "mode": "non-main",
    "scope": "agent",
    "workspaceAccess": "read-only"   // 沙箱中 workspace 只读
  }
}
```

### 2.2 Agent 不可访问的目录

以下目录对 Agent 应当是不可见或不可写的：

| 目录/文件 | 原因 | 防护方式 |
|-----------|------|----------|
| `~/.openclaw/credentials/` | 存储 API Key、OAuth Token 等凭证 | Gateway 进程读取，不暴露给 Agent |
| `/etc/`, `/usr/`, `/var/` 等系统目录 | 系统关键路径 | 沙箱 `readOnlyRoot: true` 阻止写入 |
| 其他用户 Home 目录 | 权限隔离 | Linux 用户权限 + Docker 用户隔离 |
| Docker Socket (`/var/run/docker.sock`) | 避免容器逃逸 | 沙箱默认不挂载 |
| 宿主机网络栈 | 避免内网探测 | 沙箱 `network: "none"` |

### 2.3 多 Agent 间的目录隔离

多 Agent 配置下，每个 Agent 拥有独立的 Workspace 和会话存储目录，彼此不可互相访问：

```
~/.openclaw/
├── workspace-analyst/         # Agent A 的工作区
├── workspace-architect/       # Agent B 的工作区
├── workspace-developer/       # Agent C 的工作区
└── agents/
    ├── analyst/
    │   ├── agent/auth-profiles.json   # 独立认证配置
    │   └── sessions/*.jsonl           # 独立会话记录
    ├── architect/                     # 各自隔离
    └── developer/
```

> 重要：**不要跨 Agent 复用 `agentDir`**，否则会导致认证和会话冲突，破坏隔离性。

## 3. 密钥与凭证保护

### 3.1 凭证存储机制

OpenClaw 的凭证管理遵循 **分离存储** 原则：

```
~/.openclaw/
├── credentials/               # 凭证专用目录，仅 Gateway 进程访问
│   ├── provider-keys.json     # LLM Provider API Key
│   └── channel-tokens.json    # 渠道 Bot Token
├── openclaw.json              # 配置文件中可引用环境变量，不直接写入密钥
```

**核心原则**：
- 凭证由 Gateway 进程管理，Agent Runtime 通过 Gateway API 间接调用 LLM，**不直接接触 API Key**
- Agent 发起 LLM 请求时，Gateway 负责注入认证信息，Agent 看不到实际的 Key 值

### 3.2 避免密钥明文写入配置

在 `openclaw.json` 中使用环境变量引用，而非明文密钥：

```json5
{
  // ❌ 错误：明文写入
  // "apiKey": "sk-ant-xxxxxxxx"

  // ✅ 正确：引用环境变量
  "agent": {
    "model": "anthropic:claude-sonnet-4-20250514",
    "apiKey": "${ANTHROPIC_API_KEY}"
  },

  // Webhook Secret 同样使用环境变量
  "webhooks": {
    "deploy-notify": {
      "secret": "${WEBHOOK_SECRET}"
    }
  }
}
```

环境变量可通过以下方式注入：
- **systemd 服务**：在 `.service` 文件的 `Environment=` 或 `EnvironmentFile=` 中设置
- **Docker 部署**：在 `.env` 文件或 `docker-compose.yml` 的 `environment` 中设置
- **Shell 配置**：在 `~/.bashrc` 或 `~/.zshrc` 中 `export`（注意权限）

### 3.3 Docker 部署的密钥隔离

Docker 部署时，`docker-setup.sh` 会自动生成 `.env` 文件存放 Gateway Token，避免密钥进入镜像层：

```bash
# .env 文件（不要提交到 Git）
OPENCLAW_GATEWAY_TOKEN=<auto-generated-token>
ANTHROPIC_API_KEY=sk-ant-xxxxxxxx
```

```yaml
# docker-compose.yml 引用 .env
services:
  openclaw-gateway:
    env_file: .env
    # 不要在 image 构建阶段写入密钥
```

### 3.4 Gateway Token 管理

Gateway Token 是 API 调用的认证凭证，安全管理要点：

- onboard 时自动生成，存储在配置中
- 通过 SSH 隧道传输 Dashboard URL 时附带 Token，不在公网传输
- 定期通过 `openclaw doctor` 检查 Token 配置状态
- 如怀疑泄露，重新运行 onboard 或手动重新生成

## 4. 沙箱安全详解

### 4.1 沙箱模式

沙箱通过 Docker 容器隔离 Agent 的工具执行环境：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `"off"` | 不使用沙箱，直接在宿主机执行 | 仅限可信环境的本地开发 |
| `"non-main"` | 非主会话（Cron、Webhook 触发）在沙箱中执行 | **默认推荐** |
| `"all"` | 所有会话均在沙箱中执行 | 高安全要求的生产环境 |

### 4.2 沙箱容器安全配置

```json5
{
  "sandbox": {
    "mode": "all",
    "docker": {
      "image": "openclaw-sandbox:bookworm-slim",  // 最小化基础镜像
      "readOnlyRoot": true,                        // 只读根文件系统
      "network": "none",                           // 无网络访问
      "memory": "1g",                              // 内存上限
      "cpus": 1,                                   // CPU 上限
      "setupCommand": "apt-get update && apt-get install -y git curl"  // 预装工具（可选）
    }
  }
}
```

**安全效果**：
- `readOnlyRoot: true` —— 容器内无法修改系统文件，防止恶意软件持久化
- `network: "none"` —— 无法访问外部网络，防止数据外泄和内网探测
- 资源限制 —— 防止 Agent 失控时耗尽宿主机资源

### 4.3 按 Agent 配置沙箱

不同角色的 Agent 可设置不同的沙箱策略：

```json5
{
  "agents": {
    "list": [
      {
        "id": "analyst",
        "sandbox": { "mode": "non-main" }           // 分析师：宽松
      },
      {
        "id": "developer",
        "tools": {
          "profile": "coding",
          "allow": ["exec", "write", "edit", "read"] // 开发者：允许代码操作
        }
      },
      {
        "id": "tester",
        "sandbox": {
          "mode": "all",                              // 测试：全部沙箱
          "scope": "agent"
        }
      }
    ]
  }
}
```

### 4.4 Docker 部署下的沙箱与凭证保护

当 Gateway 本身运行在 Docker 容器中时，沙箱配置需要特别考虑，因为沙箱依赖 Docker 来隔离 Agent 工具执行，而 Gateway 已经在容器内。

#### 问题：沙箱关闭时的凭证暴露

Docker 部署通常挂载整个数据卷：

```yaml
volumes:
  - openclaw_data:/home/node/.openclaw
```

该卷包含 `credentials/` 目录。如果将沙箱设为 `"off"`，Agent 工具直接在 Gateway 容器内执行，与 Gateway 进程共享文件系统，**Agent 可以读取凭证文件**。即使拆分为多个卷分别挂载，只要 Agent 和 Gateway 在同一容器内，Agent 就能访问所有挂载点。

#### 不同方案的安全对比

| 部署方式 | 沙箱模式 | Agent 能否读取 credentials | 说明 |
|----------|----------|--------------------------|------|
| 原生部署 | `"all"` / `"non-main"` | **不能** | 工具在独立 Docker 容器中执行，只挂载 workspace |
| Docker 部署 | `"off"` | **能** | 与 Gateway 同一容器，共享文件系统 |
| Docker 部署 | `"all"` + 挂载 Docker Socket | **不能** | 沙箱以兄弟容器运行，只挂载 workspace |

#### 推荐方案：挂载 Docker Socket（Sibling Container 模式）

将宿主机 Docker Socket 挂载到 Gateway 容器，让 Gateway 通过宿主机 Docker 引擎创建沙箱容器。沙箱容器作为 Gateway 的"兄弟容器"运行在宿主机上，只挂载 workspace 目录，无法访问 credentials：

```yaml
# docker-compose.yml
services:
  openclaw-gateway:
    image: openclaw-gateway:latest
    volumes:
      - openclaw_data:/home/node/.openclaw
      - /var/run/docker.sock:/var/run/docker.sock   # 挂载 Docker Socket
    # ...
```

```json5
// openclaw.json
{
  "sandbox": {
    "mode": "all",
    "docker": {
      "image": "openclaw-sandbox:bookworm-slim",
      "readOnlyRoot": true,
      "network": "none"
    }
  }
}
```

**安全分析**：

- 挂载 Docker Socket 意味着 Gateway 进程拥有宿主机 Docker 控制权
- 但 Gateway 是**确定性代码**（Node.js 应用），行为可控可审计
- Agent 是 **LLM 驱动的**，输出不确定，才是真正需要隔离的对象
- 沙箱容器由 Gateway 创建，只挂载 workspace，Agent 无法接触 credentials
- 安全文档中"不挂载 Docker Socket"指的是**沙箱容器**，不是 Gateway 容器

#### 备选方案：沙箱关闭 + 外层容器加固

如果不想挂载 Docker Socket，可以关闭沙箱并依赖外层容器提供隔离，但需接受 Agent 能访问凭证的风险：

```yaml
services:
  openclaw-gateway:
    image: openclaw-gateway:latest
    read_only: true                     # 只读根文件系统
    volumes:
      - openclaw_data:/home/node/.openclaw
    tmpfs:
      - /tmp
    networks:
      - openclaw-internal
    deploy:
      resources:
        limits:
          memory: 2g
          cpus: "2"

networks:
  openclaw-internal:
    internal: true                      # 无外网访问
```

此方案适用于完全自控的可信环境，外层容器提供了文件系统隔离、网络隔离和资源限制，但**不能阻止 Agent 读取容器内的凭证文件**。

> **总结**：Docker 部署下如需保护凭证不被 Agent 访问，应挂载 Docker Socket 并启用沙箱模式 `"all"`，让 Agent 工具在独立的兄弟容器中执行。

## 5. 网络安全

### 5.1 Gateway 访问控制

Gateway 默认绑定 `127.0.0.1`，仅接受本地连接。远程访问应通过安全隧道：

```bash
# ✅ 推荐：SSH 隧道
ssh -N -L 18789:127.0.0.1:18789 user@<server-ip>

# ✅ 推荐：Tailscale / WireGuard VPN
# Gateway 绑定 Tailscale IP，仅 VPN 内可达

# ❌ 禁止：直接暴露到公网
# "host": "0.0.0.0"  ← 不要这样做
```

### 5.2 Webhook 安全

外部系统通过 Webhook 触发 Agent 时，必须验证请求来源：

```json5
{
  "webhooks": {
    "deploy-notify": {
      "prompt": "收到部署通知...",
      "secret": "${WEBHOOK_SECRET}"    // 必须配置 Secret
    }
  }
}
```

调用时通过 `X-Webhook-Secret` Header 验证：

```bash
curl -X POST http://127.0.0.1:18789/webhook/deploy-notify \
  -H "X-Webhook-Secret: $WEBHOOK_SECRET" \
  -d '{"repo": "myapp", "message": "fix: 修复登录bug"}'
```

### 5.3 渠道认证

消息渠道（Telegram、WhatsApp 等）的 Bot Token 各自独立，配对认证（`dmPolicy: "pairing"`）确保只有经过批准的用户才能与 Agent 交互：

```bash
# 查看待配对设备
openclaw devices list

# 批准指定设备
openclaw devices approve <device-id>
```

## 6. 行为边界控制

除了系统层面的安全机制，还可以通过 Workspace 文件定义 Agent 的行为边界：

### 6.1 SOUL.md 定义安全边界

```markdown
## 边界

- 不执行破坏性操作（rm -rf、drop database 等）
- 不访问或传输敏感数据
- 不确定时主动询问确认
```

### 6.2 AGENTS.md 定义操作规则

```markdown
## 规则

- 执行命令前先确认
- 涉及文件删除或系统操作时请求授权
- 安全性 > 效率 > 便利性
```

### 6.3 工具白名单

通过 `tools.allow` 和 `tools.deny` 精确控制 Agent 可使用的工具：

```json5
{
  "tools": {
    "profile": "coding",
    "allow": ["exec", "write", "edit", "read"],   // 仅允许这些工具
    "deny": ["rm", "curl"]                          // 明确禁止的工具
  }
}
```

## 7. 安全检查与诊断

### 7.1 openclaw doctor

定期运行安全诊断，检查潜在风险：

```bash
openclaw doctor
```

检查项目包括：
- Gateway 运行状态和绑定地址安全性
- Auth Token 是否已配置
- 沙箱镜像是否存在
- DM 策略是否安全（pairing vs open）
- 凭证文件权限是否正确

### 7.2 定期安全巡检

可通过 Heartbeat 或 Cron 实现自动化安全巡检：

```markdown
## 安全巡检（HEARTBEAT.md）

- 检查 Gateway 绑定地址是否为 127.0.0.1
- 检查 ~/.openclaw/credentials/ 目录权限（应为 700）
- 检查是否有异常的配对设备
- 检查 Docker 沙箱容器是否正常运行
```

## 8. 安全实践清单

### 部署层

- [ ] Gateway 绑定 `127.0.0.1`，不暴露到公网
- [ ] 使用 Docker 部署，利用容器隔离
- [ ] 通过 SSH 隧道或 VPN 访问远程 Gateway
- [ ] `.env` 文件不提交到 Git（加入 `.gitignore`）
- [ ] 凭证目录权限设为 `700`（`chmod 700 ~/.openclaw/credentials/`）

### 配置层

- [ ] 启用沙箱模式（至少 `non-main`，生产环境推荐 `all`）
- [ ] 沙箱容器启用 `readOnlyRoot: true` 和 `network: "none"`
- [ ] API Key 使用环境变量引用，不明文写入配置
- [ ] Webhook 配置 Secret 验证
- [ ] DM 策略设为 `pairing`，不使用 `open`

### 运维层

- [ ] 定期运行 `openclaw doctor` 检查安全状态
- [ ] 多 Agent 使用独立 Workspace，不共用 `agentDir`
- [ ] 渠道 Bot Token 各 Agent 独立，互不复用
- [ ] 通过工具白名单限制 Agent 能力范围
- [ ] 在 SOUL.md 和 AGENTS.md 中明确定义行为边界
